# No Compile-Time Safety on Persona IDs in SELECTION_MATRIX

## Problem Statement

`SELECTION_MATRIX` in `persona-selector.ts` is typed as `Record<TableTemperature, string[]>` — the values are plain strings. A typo in any persona ID (e.g. `"exploit_haw"`) compiles silently and produces an empty candidates list at runtime, silently falling back to GTO Grinder with no warning. Similarly, `FALLBACK_ID = "gto_grinder"` is an unguarded string constant.

`personas.ts` is auto-generated by `scripts/generate-charts.ts`. If the generator renames a persona ID, both `SELECTION_MATRIX` and `FALLBACK_ID` become stale with no compile error.

The `TYPE_MAP` in `table-temperature.ts` has the same maintenance obligation: its keys (`TIGHT_PASSIVE` etc.) must stay in sync with the `inferredType` strings emitted by `lib/ai/schema.ts`, but there is no compiler enforcement of this link.

## Findings

- `lib/poker/persona-selector.ts:32-39` — `SELECTION_MATRIX: Record<TableTemperature, string[]>` — plain string arrays, no ID validation
- `lib/poker/persona-selector.ts:42` — `const FALLBACK_ID = "gto_grinder"` — untyped string literal
- `lib/poker/personas.ts:1-7` — generated file has no exported `PersonaId` type union that consumers could reference
- `lib/poker/table-temperature.ts:29-34` — `TYPE_MAP` keys are AI schema `inferredType` values; no comment links them to `lib/ai/schema.ts`
- Code quality review (2026-02-23): rated MEDIUM

## Proposed Solutions

### Option A: Add `PersonaId` type to personas.ts and use it (Recommended)

Add to `personas.ts` (or keep it manually in a types file since personas.ts is auto-generated):

```typescript
// In lib/poker/persona-selector.ts or a separate types file:
export type PersonaId = "gto_grinder" | "tag_shark" | "lag_assassin" | "exploit_hawk";
```

Then:
```typescript
const FALLBACK_ID: PersonaId = "gto_grinder";
const SELECTION_MATRIX: Record<TableTemperature, PersonaId[]> = { ... };
```

TypeScript will error immediately if any ID string doesn't match the union.

**Effort:** Small — add type alias, update two type annotations
**Risk:** None

### Option B: Derive PersonaId from PERSONAS array

```typescript
// Auto-derive from the generated array (type-level)
export type PersonaId = typeof PERSONAS[number]["id"]; // "gto_grinder" | ...
```

This keeps the type in sync with the generated file automatically.

**Effort:** Small — one line in personas.ts (or via generate-charts.ts)
**Risk:** None — purely additive

### Option C: Add a comment to TYPE_MAP linking to schema.ts

At minimum, add:
```typescript
// Keys must match inferredType values in lib/ai/schema.ts → Opponent.inferredType
const TYPE_MAP: Record<string, TableTemperature> = { ... }
```

**Effort:** One comment
**Risk:** None — convention only, no compile enforcement

## Recommended Action

Option B for `PersonaId` (derive from generated array, stays in sync automatically) + Option C for `TYPE_MAP` comment (no way to auto-enforce schema.ts link without a separate codegen step).

## Technical Details

- **Affected files:** `lib/poker/personas.ts` or `lib/poker/persona-selector.ts`, `lib/poker/table-temperature.ts`
- **Scripts:** `scripts/generate-charts.ts` could be updated to emit the `PersonaId` type
